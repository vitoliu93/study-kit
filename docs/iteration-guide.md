# study-kit 自我迭代指南

> 本插件如何改自己。改任何机制前先读 north-star.md,再读本文。

## 迭代原则(按序裁决,过不了就不改)

1. **北极星裁决**:凡增加下次学习会话启动成本(操作或情绪)的变更,一律否决。
2. **使用失败驱动 > 灵感驱动**:每个变更必须能指认触发事件——哪一次存档/读档/审计
   暴露了什么缺陷。"我想到一个好功能"不是触发事件,按单主线红线处理:先过目标审计。
3. **伴随学习行为**:没有学习会话发生的日子不动工具。工具建设是最高级的拖延。
4. **插件装机制,项目装目标**:任何具体学习目标/材料/路线不得写进插件;发现硬编码即为缺陷。
5. **原子技能优先复用**:能连用 grill-me / blindspot / skill-atlas 的不重造,
   但一律软依赖 + 内置降级路径,保证插件独立可用。

## 变更流程

1. 记录触发事件(一句话:哪次使用、什么失败)。
2. 过上面五条原则;过不了就地否决,不留 backlog——backlog 是囤积的另一种形态。
3. 最短 diff 修改;机制变化 bump minor,文案/修复 bump patch。
4. commit message 首行引用触发事件。

## 体检与红队

- **改 description 后**:跑 `/skill-atlas`(route-overlap + trigger eval),
  重点盯 resume-learning 的"继续/存档"类触发词与 handoff("continue/接手")的路由边界。
- **红队(机会主义,不设计数器——没人会去数存档次数,数字节律是死条款)**:
  每次 minor 版本发布前必跑一轮;平时想起来就跑。用 ask-ai(clean context,
  opus + high)做对抗:攻击面固定两个——
  **有效性**(机制会不会没人用、被绕过、被玩坏:如为刷频率指标开水会话)、
  **可行性**(机制在真实约束下能否落地:作用域、触发路由、软依赖可达性)。
  红队发现按本指南流程处置,不直接改。
- 红队和体检自身也受原则 3 约束:学习都停摆的月份,先救学习,不搞体检。

## 反目标(永不迭代成的样子)

- 不长成学习管理系统(计划表/仪表盘/打卡)——那是给自律者的杠杆,对拖延症是晒网理由。
- 不加第二条 L3 红线。
- 不做成需要维护的东西——维护成本也是启动成本。
